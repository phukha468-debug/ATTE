# ADR-002: Multi-tenant RLS Policies
**Дата:** 14.04.2026
**Статус:** accepted

## Контекст
База данных хранит B2B-информацию нескольких компаний. Включен Row Level Security (RLS). Нам нужно гарантировать криптографическую изоляцию: сотрудники видят только свои данные, менеджеры видят данные своего отдела, вопросы видят все авторизованные.

## Решение
Мы используем встроенную функцию Supabase `auth.uid()` для идентификации пользователя. 
Политики (Policies):
1. **`questions`**: `SELECT` доступен всем авторизованным. `INSERT/UPDATE` — только через админку (Service Role).
2. **`users`**: Пользователь видит свой профиль (`id = auth.uid()`). Менеджеры/Админы видят профили всех пользователей, у которых совпадает `company_id`.
3. **`companies`**: Доступ на чтение только если `id` компании совпадает с `company_id` текущего пользователя.
4. **`test_results`**: Пользователь может делать `INSERT` только для своего `user_id`. Чтение — только свои результаты. Менеджеры могут читать все результаты внутри своей `company_id`.
5. **`payments`**: Только для менеджеров и админов своей компании.

## Key Map
- Клиент (TMA) использует `anon` ключ. Запросы автоматически фильтруются на уровне ядра PostgreSQL согласно политикам выше. Никакой логики фильтрации на фронтенде не требуется.
- Скрипты наполнения (Seed) используют `service_role` ключ для обхода этих правил.

## Антипаттерны
- ❌ Использование `service_role` ключа на фронтенде для обхода проблем с RLS.
- ❌ Фильтрация данных на фронтенде (`.filter()`) вместо полагания на RLS БД.
