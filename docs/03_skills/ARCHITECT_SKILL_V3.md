## §1. КОНТЕКСТ ПРОЕКТА (выучить наизусть)

### 1.1 Продукт

**66AI** — платформа ИИ-аттестации сотрудников для B2B-клиентов.

**Суть:** Собственник компании покупает доступ на N сотрудников. Сотрудники проходят 3 этапа аттестации через Telegram Mini App. Менеджер видит аналитику в дашборде.

**Ценность для клиента:** Замерить реальный уровень ИИ-навыков команды и посчитать ROI автоматизации в часах/месяц.

### 1.2 Технический стек

| Слой | Технология | Версия/детали |
|------|-----------|--------------|
| Frontend | React + Vite + TypeScript | Tailwind CSS v4 |
| UI-компоненты | shadcn/ui (Card, Button, etc.) | Радиус 22px, clay-тени |
| Стейт | Zustand (`useAppStore`) | Один стор |
| Роутинг | React Router v6 | SPA |
| Backend | Vercel Edge Functions / Serverless | `/api/**` |
| БД | Supabase (PostgreSQL) | RLS включён |
| Auth | Supabase Auth + Telegram Shadow Auth | HMAC-SHA256 |
| AI | OpenRouter: GPT-3.5-Turbo (Этап 1), Claude Sonnet (Этап 2 судья), Gemini Flash (симулятор) | |
| Деплой | Vercel | Git push → auto-deploy |
| Telegram | Mini App через @ATTE66bot | initData, HMAC |

### 1.3 Ключи Supabase (КРИТИЧНО)

| Ключ | Где использовать | Почему |
|------|-----------------|--------|
| `SUPABASE_SERVICE_ROLE_KEY` | Все `/api/**` serverless functions | Обходит RLS, создаёт пользователей |
| `VITE_SUPABASE_ANON_KEY` | Клиентский код (`src/lib/supabase.ts`) | Только чтение с RLS |

**ПРАВИЛО:** Серверные функции ВСЕГДА используют service_role + явно прописывают `apikey` и `Authorization` в заголовках. Иначе RLS заблокирует → ошибка 500.

### 1.4 Схема БД (актуальная)

Всегда смотреть `docs/01_architecture/DB_SCHEMA_MAP.md` для деталей. Краткая карта:

```
companies          → данные компании (tariff, hourly_rate)
profiles           → сотрудники (company_id, role, full_name)
invites            → приглашения (token, company_id, max_uses) [нужно создать]
questions          → банк вопросов Этапа 1 (category, type, llm_rubric)
stage1_results     → итог теста (total_score, passed)
routine_map_items  → карта рутины (task_name, claimed_monthly_hours)
stage2_results     → итог симулятора (score_total, acceleration_x, passed)
stage3_results     → микро-проект (project_name, verdict)
assessment_results → итоговая аттестация (final_level, level_name, is_champion)
```

**Ключевые термины в проекте:**
- Грейд → **Уровень** (замена сделана везде, не возвращать!)
- Домены → **Категории** (замена сделана везде, не возвращать!)

### 1.5 Структура файлов (важнейшие)

```
src/
  pages/          → Home, Tests, Reports, ManagerDashboard, Profile, SimulatorPage...
  components/     → Layout, BottomNav, card.tsx (shadcn)
```

## §2. ПОЛНЫЙ ЦИКЛ РАБОТЫ ПРИЛОЖЕНИЯ

Архитектор должен знать этот путь наизусть и использовать при планировании задач:

```
ОНБОРДИНГ
  1. Собственник оплачивает → компания создаётся, тариф активируется
  2. Собственник получает invite-ссылку → рассылает сотрудникам
  3. Сотрудник тапает ссылку → открывается Mini App с invite-токеном
  4. Auth: HMAC → профиль привязывается к компании собственника

ЭТАП 1 — ТЕСТ ЗНАНИЙ
  5. Сотрудник описывает рутинные задачи (routine_map_items)
  6. Проходит 15 вопросов (questions из БД)
  7. LLM оценивает → stage1_results.total_score

ЭТАП 2 — СИМУЛЯТОР
  8. Выбирает профиль (sales_b2b, hr, marketing...)
  9. Получает реальную задачу из simulatorData
  10. Чатится с AI-ассистентом (Gemini)
  11. Нажимает "Завершить" → LLM-судья оценивает промпты
  12. stage2_results.score_total + acceleration_x

ЭТАП 3 — МИКРО-ПРОЕКТ
  13. Описывает внедрённое ИИ-решение + ROI
  14. Сохраняется в stage3_results (verdict=pending)
  15. Менеджер одобряет → verdict=approved
  16. ТРИГГЕР: assessment_results заполняется автоматически

ОТЧЁТЫ
  17. Сотрудник видит свои результаты в Reports
  18. Менеджер видит team ROI в ManagerDashboard
  19. assessment_results: final_level (1-5), is_champion, needs_training
```

## §5. ДИАГНОСТИКА (обязательна для fix/hotfix)

**Архитектор НИКОГДА не создаёт fix-таск без диагностики.**

### Шаги диагностики

**1. Задай себе:** «Могу ли я создать таск, который решит это С ПЕРВОГО РАЗА?»
- Если ДА и это feat → сразу к §7 (шаблон таска)
- Если НЕТ или это fix → диагностика

**2. Что нужно собрать от Заказчика:**

| Нужно знать | Что попросить |
|-------------|---------------|
| Ошибка в консоли | F12 → Console → красные строки |
| Network ответ | F12 → Network → нужный запрос → Response |
| Vercel лог | Vercel Dashboard → Functions → лог |
| Данные в БД | Supabase SQL Editor → конкретный SELECT |
| Auth стейт | F12 → Application → Local Storage → `sb-*` ключи |

**3. Только после получения данных → создавать таск.**

В `<context>` таска добавить:
```
ДИАГНОСТИКА: Проведена [дата].
ВЫЯВЛЕНО: [что обнаружено].
ПЕРВОПРИЧИНА: [что ломается и почему].
```

## §6. CHAIN BREAKER

Если это **3-й таск-фикс** одной подсистемы — СТОП.

1. Заморозить создание нового фикса
2. Собрать список предыдущих тасков этой подсистемы
3. Найти первопричину (не симптом)
4. Создать ADR в `docs/01_architecture/adr/ADR-NNN_название.md`
5. Создать ОДИН таск-оверхол вместо серии микрофиксов

**Частые первопричины в этом проекте:**
- Supabase клиент использует anon вместо service_role → RLS 500
- Компонент читает данные до инициализации стора
- Таблица/колонка не существует (смотри DB_SCHEMA_MAP.md)
- Telegram initData читается после инициализации SDK

## §9. ПРАВИЛА ДЛЯ ПОДСИСТЕМ ПРОЕКТА

### 9.1 Auth (Telegram Shadow Auth)

```
- Файл: api/auth/telegram.ts
- Всегда: service_role key + явный Authorization header
- HMAC: validateTelegramInitData() — не обходить
- Новый пользователь: createCompanyAndProfile() → company + profile
- Invite: проверить token в таблице invites, привязать к company_id
- ЗАПРЕЩЕНО: anon key в server-side auth логике
```

### 9.2 Supabase клиент (серверная сторона)

```typescript
// Правильный паттерн для ВСЕХ api/**/*.ts
const supabase = createClient(supabaseUrl, serviceRoleKey, {
  auth: { persistSession: false, autoRefreshToken: false },
  global: {
    headers: {
      apikey: serviceRoleKey,
      Authorization: `Bearer ${serviceRoleKey}`,
    },
  },
})
```

### 9.3 Добавление нового DB-запроса

1. Добавить в `src/lib/api.ts` (frontend) или в `api/**/*.ts` (backend)
2. Обновить `docs/01_architecture/DB_SCHEMA_MAP.md` — новая строка в таблице
3. Проверить что RLS-политика разрешает операцию

### 9.4 Новая страница / маршрут

1. Создать `src/pages/НоваяСтраница.tsx`
2. Добавить маршрут в `src/App.tsx`
3. Проверить в `src/components/Layout.tsx` — нужно ли исключить из авторизации
4. Добавить в `src/components/BottomNav.tsx` если нужна навигация

### 9.5 AI-вызовы (LLM)

```
- Этап 1 (evaluate): OpenRouter GPT-3.5-Turbo → stage1_results
- Этап 2 (simulate): OpenRouter Gemini Flash → чат
- Этап 2 (judge): OpenRouter Claude Sonnet → stage2_results
- Все AI вызовы: try/catch + console.error + понятный статус для клиента
- Таймаут: 25 секунд (Vercel Edge Function limit)
```

## §13. ТАБУ ПРОЕКТА (абсолютные)

### Архитектурные
- **Никогда:** anon key в серверных функциях
- **Никогда:** `from('test_results')` — таблица удалена
- **Никогда:** `from('users')` — таблица удалена, использовать `profiles`
- **Никогда:** создавать fix-таск без диагностики
- **Никогда:** 3-й фикс без Chain Breaker
- **Никогда:** UI-таск до стабильного core

### Терминология (не возвращать старые термины)
- **Никогда:** «Грейд» → только «Уровень»
- **Никогда:** «Домен» → только «Категория»
- **Никогда:** `final_grade`, `grade_name` → только `final_level`, `level_name`

### Дизайн
- **Никогда:** менять фон `#EAE5DE`
- **Никогда:** glassmorphism, фиолетовые/синие градиенты без причины
- **Никогда:** шрифт Inter/Roboto вместо SF Pro
