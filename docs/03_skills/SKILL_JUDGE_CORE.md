# STAGE 2 JUDGE — CORE SKILL

## НАЗНАЧЕНИЕ

Этот скилл — ядро оценки Этапа 2. Он анализирует, КАК сотрудник использовал ИИ для решения рабочей задачи, и генерирует развёрнутую обратную связь, которая не просто ставит баллы, а показывает, что делать лучше.

Judge получает на вход две вещи:
1. **Лог переписки** из Sandbox (все промпты сотрудника и ответы ИИ-ассистента)
2. **Контекст** — задание, бенчмарк, профиль должности, данные Этапа 1

Judge НЕ общается с сотрудником. Он работает постфактум, анализируя полную картину.

## 5 ИЗМЕРЕНИЙ ОЦЕНКИ

### Измерение 1: КАЧЕСТВО ПРОМПТИНГА (вес 30%)

Анализируй ВСЮ цепочку промптов сотрудника. Не один промпт, а паттерн.

**Шкала 1-5:**

| Балл | Описание | Индикаторы |
|------|----------|-----------|
| 1 | Не владеет | Промпт из 1-2 слов: «напиши КП», «сделай письмо». Нет контекста, роли, формата. |
| 2 | Базовый | Есть тема, но нет деталей. «Напиши КП для клиента на CRM-систему». Нет роли, формата, ограничений. |
| 3 | Средний | Есть контекст и какая-то структура: указан клиент, продукт, но нет формата вывода или роли. Или есть роль, но нет контекста. |
| 4 | Хороший | Полноценный промпт: роль + контекст + данные из брифа + формат + ограничения. Минимум 1 итерация. |
| 5 | Отличный | Промпт-паттерн эксперта: чёткая роль, весь контекст из задания, конкретный формат, примеры, ограничения. Несколько итераций с точными доработками. Результат предыдущего промпта используется в следующем. |

**Что именно проверять в логе:**
- Передал ли сотрудник в промпт данные из брифа задания? (Если задание содержит ТЗ клиента с бюджетом 500 000 — упомянут ли бюджет в промпте?)
- Указана ли роль ИИ?
- Указан ли формат вывода?
- Указаны ли ограничения (объём, тон, что не включать)?
- Есть ли структура промпта (не поток сознания)?

---

### Измерение 2: ИТЕРАТИВНОСТЬ (вес 20%)

Это ключевой индикатор зрелости. Новичок принимает первый ответ. Эксперт дорабатывает.

**Шкала 1-5:**

| Балл | Описание | Индикаторы |
|------|----------|-----------|
| 1 | Нет итераций | 1 промпт → принял результат как есть. |
| 2 | Формальная итерация | 2 промпта, но второй — «сделай лучше» / «переделай». Нет конкретных указаний. |
| 3 | Базовая доработка | 2-3 промпта с конкретными правками: «добавь раздел про гарантии», «измени тон на более деловой». |
| 4 | Целенаправленная доработка | 3+ промпта. Сотрудник проверяет результат, находит проблемы, формулирует точные доработки. Видна логика улучшения. |
| 5 | Экспертный пайплайн | Каждый промпт — осознанный шаг. Результат предыдущего = вход следующего. Сотрудник строит цепочку: сначала структуру, потом наполнение, потом шлифовку. |

**Что именно проверять:**
- Количество промптов (но не механически — 10 промптов «переделай» хуже 3 точных)
- Ссылается ли сотрудник на конкретные части предыдущего ответа?
- Есть ли прогресс от промпта к промпту?
- Использует ли результат шага N в шаге N+1?

---

### Измерение 3: УСКОРЕНИЕ (вес 20%)

Ускорение = Бенчмарк / Фактическое время. Но с поправкой на качество.

**Шкала:**

| Ускорение | Балл | Интерпретация |
|-----------|------|--------------|
| < 1.5x | 1 | Практически не ускорился. ИИ не помог или мешал. |
| 1.5-2.5x | 2 | Слабое ускорение. Использовал ИИ, но неэффективно. |
| 2.5-5x | 3 | Умеренное ускорение. Адекватное использование. |
| 5-10x | 4 | Хорошее ускорение. Грамотная работа с ИИ. |
| > 10x | 5 | Экспертное ускорение. Возможно только при отличных промптах. |

**Важная поправка:**
Если ускорение высокое, но качество результата низкое (измерение 4 < 3) — снизить балл ускорения на 1. Быстро и плохо ≠ эффективно.

Если ускорение низкое, но качество отличное (измерение 4 ≥ 4) — не снижать. Возможно, сотрудник тщательно проверял результат, и это хорошо.

---

### Измерение 4: КАЧЕСТВО РЕЗУЛЬТАТА (вес 20%)

Здесь Judge использует **профильный модуль** — критерии, специфичные для должности и задачи. Core Skill определяет структуру, профильный модуль — содержание.

**Общая шкала:**

| Балл | Описание |
|------|----------|
| 1 | Результат нерелевантный или непригодный. Generic текст, не связан с заданием. |
| 2 | Частично пригоден. Есть структура, но нет персонализации, пропущены ключевые блоки. |
| 3 | Пригоден с доработкой. Основное на месте, но нужна ручная правка (20-30% текста). |
| 4 | Хороший. Можно использовать с минимальными правками (5-10%). |
| 5 | Отличный. Готов к использованию «как есть» или с косметическими правками. |

**Как оценивать:**
1. Загрузи профильный модуль для данного profile_id (напр. `../02_tests/stage2/profiles/PROFILE_SALES_B2B.md`, раздел 7)
2. Проверь финальный результат по каждому профильному критерию
3. Суммируй профильные баллы → преобразуй в шкалу 1-5

Преобразование: профильный балл / максимум × 5, округлить.
Пример: КП набрало 7/10 по профильным критериям → 7/10 × 5 = 3.5 → округляем до 4.

---

### Измерение 5: ПОНИМАНИЕ ОГРАНИЧЕНИЙ ИИ (вес 10%)

Проверяем: не принял ли сотрудник потенциальные галлюцинации за факт?

**Шкала 1-5:**

| Балл | Описание |
|------|----------|
| 1 | Попросил ИИ то, что ИИ не может (реальные данные, базы email, прогнозы рынка как факт). Принял всё без проверки. |
| 2 | Использовал ИИ корректно, но не проверил факты в результате. Или результат содержит очевидную галлюцинацию, которую сотрудник не заметил. |
| 3 | Использовал ИИ в рамках его возможностей. Результат без грубых галлюцинаций, но сотрудник не демонстрирует осознанной проверки. |
| 4 | Явно проверяет результат: просит уточнить источники, сам дополняет реальными данными, исправляет неточности. |
| 5 | Демонстрирует экспертное понимание: разделяет, что ИИ может и не может, сам вносит реальные цифры/факты, использует ИИ только для генерации/структуры. |

**Красные флаги (автоматическое снижение до 1-2):**
- Попросил ИИ «найти контакты клиента» или «показать данные из CRM»
- Принял выдуманные цифры (доля рынка, статистика) как реальные
- Не проверил юридические нормы, сроки, законы в сгенерированном тексте

## ФОРМУЛА ИТОГОВОГО БАЛЛА

```
Итоговый балл = (Промптинг × 0.30) + (Итеративность × 0.20) + 
                (Ускорение × 0.20) + (Качество × 0.20) + 
                (Ограничения × 0.10)

Шкала: 1.0 - 5.0
```

**Маппинг на уровни:**

| Балл Stage 2 | Интерпретация | Уровень (с учётом Stage 1) |
|-------------|---------------|-------------------------|
| < 2.0 | Слабо. ИИ не помог. | Уровень 1-2 |
| 2.0 - 2.9 | Базовое использование. | Уровень 2 |
| 3.0 - 3.4 | Умеренное. Есть навыки, но неровные. | Уровень 2-3 |
| 3.5 - 3.9 | Хорошее. Уверенный пользователь. | Уровень 3 |
| 4.0 - 4.4 | Сильное. Системный подход. | Уровень 3-4 |
| 4.5 - 5.0 | Экспертное. Пайплайны, проверка, масштаб. | Уровень 4-5 |

## ГЕНЕРАЦИЯ ОБРАТНОЙ СВЯЗИ

Это ГЛАВНЫЙ выходной продукт Judge. Не баллы, а конкретные рекомендации.

### Структура обратной связи:

**Блок 1: Результат (краткий)**
- Итоговый балл, время, ускорение
- 1 предложение: что получилось хорошо

**Блок 2: Анализ промптов (подробный)**
- Для КАЖДОГО промпта сотрудника: что хорошо, что можно улучшить
- Если промпт слабый — показать, как бы выглядел лучший вариант (взять из эталонных промптов профиля)
- Конкретные формулировки, а не абстрактные советы

**Блок 3: Главная зона роста**
- Определи 1-2 самых критичных проблемы из 5 измерений
- Для каждой: проблема → почему это важно → что делать → пример

**Блок 4: Связка с картой рутины (Этап 1)**
- Сопоставь задание симулятора с задачей из карты рутины
- Рассчитай реалистичную экономию на основе показанного ускорения
- Формулировка: «Вы указали, что тратите X часов на [задачу]. На симуляторе вы показали ускорение Yx. Это означает экономию Z часов/неделю = W ₽/месяц.»

**Блок 5: Рекомендации**
- 2-3 конкретных действия
- Для уровеньа 1-2: ссылка на обучающий модуль
- Для уровеньа 3: конкретные техники (Role+Context+Format, итеративный промптинг)
- Для уровеньа 4-5: масштабирование, шаблоны, наставничество

### Правила генерации:

1. **Конкретность.** Не «улучшите промптинг». А «в вашем промпте не было контекста клиента — добавьте: отрасль, размер компании, боль. Пример: [конкретный промпт]».

2. **Честность без жёсткости.** Если сотрудник сделал плохо — скажи прямо, но конструктивно. «Ваш промпт "напиши КП" — слишком короткий. ИИ не телепат. Вот как выглядит промпт, который даёт результат с первого раза: [пример]».

3. **Связка с реальной работой.** Каждый комментарий должен отвечать на вопрос сотрудника: «И что мне с этим делать завтра на работе?»

4. **Валидация Этапа 1.** Если сотрудник заявил высокий потенциал экономии на Этапе 1, но показал слабый промптинг на Этапе 2 — отметить расхождение мягко: «Потенциал есть, но нужно прокачать навык промптинга, чтобы его реализовать».

## ИНТЕГРАЦИЯ С ЭТАПОМ 3

Данные Judge передаются в Этап 3:
- **Задание для микро-проекта** подбирается из карты рутины, но с учётом результатов Этапа 2. Если сотрудник показал хорошее ускорение на КП — микро-проект может быть «внедрить шаблон КП в отдел».
- **Зоны роста** из обратной связи → рекомендации в финальном отчёте.
- **Баллы** → расчёт уровеньа (уровень 3 требует Этап 1 ≥ 75% + Этап 2 ≥ 70%).

## ИСПОЛЬЗОВАНИЕ ПРОФИЛЬНЫХ МОДУЛЕЙ

Core Skill вызывает профильный модуль так:

1. Получи `profile_id` из входных данных
2. Загрузи файл профиля: `../02_tests/stage2/profiles/PROFILE_{profile_id}.md`
3. Из профиля возьми:
   - Раздел 4 (типичные ошибки) — для диагностики промптов
   - Раздел 3 (эталонные промпты) — для обратной связи
   - Раздел 7 (профильные критерии качества) — для измерения 4
   - Раздел 2 (ТОП-3 задачи для автоматизации) — для рекомендаций
4. Оценку промптинга, итеративности и ускорения — делай по Core Skill
5. Оценку качества результата — делай по профильным критериям
